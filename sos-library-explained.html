1

The idea: one button, no HTML on the page

HTML

Most widgets ask you to paste a chunk of HTML onto every page that uses them (the modal markup, the buttons, the footer). That's fragile — if you update the markup, every page that copy-pasted it goes out of date.

Instead, the SOS library only asks the page for one thing that already exists: a trigger button.

<button id="btnSOS">SOS</button>
Why: every other page just needs that one line, plus SOS.init(...). The entire modal — header, grid, footer — is created by JavaScript the first time it's opened, so there's exactly one place that markup can go wrong: sos.js itself.
2

Splitting the files (separation of concerns)

Architecture

Four files, four jobs. This is a very old, very useful idea in software: each file should have one reason to change.

sos.jsAll behavior. DOM creation, state, events, networking.
sos.cssAll appearance. Colors, spacing, animation.
sos-flow.jsonThe backend. Two HTTP endpoints + SQL.
sos-demo.htmlAn example of gluing it all together.
Why: a designer can re-theme the whole widget by editing only sos.css's :root block, without ever opening sos.js — and vice versa. That's the payoff of the split.
3

CSS custom properties (design tokens)

CSS

Instead of writing #E63950 forty times across the file, we write it once, name it, and refer to the name everywhere else.

:root {
  /* declare it once */
  --sos-red: #E63950;
}

.sos-send {
  /* use the name everywhere */
  background: linear-gradient(135deg, var(--sos-red), var(--sos-red-dark));
}

We go one step further and give every token a fallback:

--sos-shadow: var(--modal-shadow, 0 24px 64px rgba(0,0,0,.6));
/*            ^ read the host page's own brand token first
                if it's missing, use this literal value instead */
Why: on an ERP page that already has brand.css loaded, SOS automatically matches the house style (same shadow depth, same border radius). On a page that doesn't have it, the fallback keeps SOS looking correct anyway. One rule, two situations.
4

The IIFE + closure — why SOS is safe

JavaScript

The whole file is wrapped in a function that runs itself immediately:

// IIFE = Immediately Invoked Function Expression
const SOS = (() => {
  'use strict';
  let config = { ... };   // private — nobody outside can touch this
  let state  = { ... };   // private too

  const init = (options) => { ... };
  const destroy = () => { ... };

  return { init, destroy, open, close };  // only these 4 are public
})();

Everything declared inside — config, state, every helper function — lives in a private bubble called a closure. Only the four things in the final return are visible outside as SOS.init(), SOS.destroy(), etc.

Why: a page can't accidentally do SOS.state.selectedAlert = 'hacked' from the console — that property simply doesn't exist outside the closure. It's how the library protects itself from every other script on the page, including itself being loaded twice.
5

One state object to describe "right now"

JavaScript

Instead of scattering booleans and variables everywhere, every fact about the widget's current moment lives in one plain object:

let state = {
  alerts: [],            // what the server returned
  selectedAlert: null,    // which button is picked
  selectedIsCustom: false,// is it the "Other" reason?
  isLoading: false,        // GET in progress?
  isSending: false,        // POST in progress?
  cacheTimestamp: 0,       // when we last fetched
  abortController: null,   // how to cancel a fetch
};
Why: every function that needs to decide something ("can Send be clicked?") reads from this one object instead of five different variables. It also makes SOS.destroy() trivial — resetting the whole widget is just replacing this one object.
6

Building the modal with JavaScript, not HTML

DOM

The very first time someone opens the modal, this function runs once and builds the whole thing:

const createModal = () => {
  const overlay = document.createElement('div');
  overlay.className = 'sos-overlay';
  overlay.innerHTML = `
    <div class="sos-modal">
      <div class="sos-header">...</div>
      <div class="sos-grid"></div>  <!-- filled in later -->
      <div class="sos-footer">...</div>
    </div>`;
  document.body.appendChild(overlay);

  els.overlay = overlay;                          // keep a reference
  els.grid = overlay.querySelector('.sos-grid');   // so we can find it fast later
};

Every later openModal() call checks if (!els.overlay) createModal(); — so the expensive "build the DOM" step only ever happens once, no matter how many times the user opens and closes it.

7

A render function for every possible state

JavaScript

The little grid of reason buttons can be in exactly one of four situations at any moment. We write one function per situation, and one line of logic decides which one runs:

renderLoading()Spinner + "Loading…" while the GET is in flight.
renderAlerts()The real buttons, once data arrives.
renderEmpty()Server responded, but with zero reasons.
renderError()The request failed — shows a Retry button.
Why: this is far easier to reason about than one giant function with lots of ifs scattered through it. Each function does exactly one job, and you can test/read them in isolation.
8

Event binding & delegation

JavaScript

The grid of reason buttons is thrown away and rebuilt every time the list changes. If we attached a click listener to each button, we'd have to re-attach it every single re-render — messy, and easy to leak. Instead, we listen once on the parent:

const onGridClick = (e) => {
  const btn = e.target.closest('.sos-alert');  // "which button, if any?"
  if (btn) selectAlert(btn);
};

els.grid.addEventListener('click', onGridClick);  // bound ONCE, forever
Why: this is called event delegation — one listener on a stable parent handles clicks on any number of children, even ones that don't exist yet at the time you wrote the listener.
9

Accessibility: the focus trap & ARIA

Accessibility

Two problems every modal has to solve for someone using only a keyboard or a screen reader:

1. Tab shouldn't escape the modal. We find every focusable element inside it, and if Tab would move focus past the last one, we loop back to the first:

if (e.key === 'Tab') {
  const focusable = els.modal.querySelectorAll(
    'button:not([disabled]), input:not([disabled])'
  );
  const first = focusable[0];
  const last  = focusable[focusable.length - 1];
  if (e.shiftKey && document.activeElement === first) { e.preventDefault(); last.focus(); }
  else if (!e.shiftKey && document.activeElement === last) { e.preventDefault(); first.focus(); }
}

2. Screen readers need to know what's on screen without seeing it. That's what these attributes are for:

<div class="sos-modal" role="dialog" aria-modal="true" aria-labelledby="sos-title">
<div class="sos-state" role="status" aria-live="polite">   <!-- loading -->
<div class="sos-state" role="alert">                       <!-- error -->
Why: aria-live="polite" makes a screen reader announce "Loading alert reasons…" the instant it appears, without the user having to go looking for it. role="dialog" tells assistive tech "everything outside this is temporarily irrelevant."

Talking to the server: fetch, apiCall, and CORS

Networking

There are two different situations the library has to handle:

Same origin as the ERPUse the ERP's own apiCall() helper, so login/session cookies work exactly like every other request in the app.
Different origin(e.g. a local Node-RED server on a different port) — use a plain fetch() designed to never trigger a CORS "preflight" the backend can't answer.
const request = async (url, options = {}) => {
  if (typeof apiCall === 'function' && !isCrossOrigin(url)) {
    return apiCall(url, options);   // same-origin: use the ERP's helper
  }
  return simpleFetch(url, options); // cross-origin: avoid a CORS preflight
};
Why: a browser sends a "preflight" (an OPTIONS request) before certain cross-origin requests, to ask the server for permission first. Some simple backends (like a bare Node-RED flow) can't answer that. So for cross-origin calls we keep the request in the browser's "simple request" category — plain text body, no custom headers — which skips the preflight entirely.
11

async/await + try/catch/finally

Networking

Every network call follows the exact same shape:

const fetchAlerts = async () => {
  state.isLoading = true;
  renderLoading();                 // 1. show something immediately

  try {
    const data = await request(url, { method: 'GET' });
    state.alerts = data.alerts;
    renderAlerts();                // 2. success
  } catch (err) {
    renderError(err.message);      // 3. failure
  } finally {
    state.isLoading = false;       // 4. ALWAYS runs, success or not
  }
};
Why finally matters: without it, a thrown error could leave isLoading stuck as true forever, permanently disabling the UI. finally guarantees the cleanup step runs no matter which path the code took.
12

Conditional UI: the "Other" text box

JavaScript

This feature is just three small decisions, chained together:

user clicks a button→ is its label "Other"?→ yes: reveal textbox / no: hide + clear it 
const isOtherAlert = (label, custom) =>
  custom === true || label?.trim().toLowerCase() === 'other';

if (isOtherAlert(state.selectedAlert, state.selectedIsCustom)) {
  showOtherInput();   // reveal + focus the <input>
} else {
  hideOtherInput();   // hide it AND clear its value
}

Then, on Send, one more decision: what text do we actually submit?

let alertText = state.selectedAlert;             // default: the button's own label
if (isOtherAlert(state.selectedAlert, state.selectedIsCustom)) {
  alertText = otherInput.value.trim();            // override: whatever they typed
  if (!alertText) { notify('Please enter a reason.'); return; } // guard
}
Why hide-and-clear together: if we only hid the box without clearing it, picking "OIL STAIN" after previously typing something in "Other" would leave that old text sitting in a hidden input — invisible, but still there if the user went back to "Other".
13

Caching, and cancelling a request in flight

Performance

Caching — don't re-ask the server for the same list every time the modal opens; remember when we last asked, and skip the network if it was recent enough:

const isFresh = (Date.now() - state.cacheTimestamp) < config.cacheDuration;
if (isFresh) { renderAlerts(); return; }  // reuse what we already have

Cancelling — if the user opens, then quickly closes, the modal while a request is still travelling to the server, we tell the browser to abandon it, so it can't come back later and update DOM that no longer exists:

state.abortController = new AbortController();
request(url, { signal: state.abortController.signal });

// later, on close:
state.abortController.abort();  // the fetch above rejects immediately
Why: without this, a slow request that finishes after the user already closed the modal could still try to render into elements that are gone, or flash old data onto a screen the user has moved away from.
14

Designing the public API

API design

Everything the outside world can do to the widget is exactly four verbs:

SOS.init(options)Wire the trigger button and store config. Refuses to run twice.
SOS.open() / SOS.close()Programmatic control, in case a page wants a keyboard shortcut.
SOS.destroy()Undo everything init() did — for single-page apps that mount/unmount views.

New features (like caching or the debug flag) were added as optional keys on the same object, never as new required arguments:

// old integration code — still works, unchanged:
SOS.init({ button: '#btnSOS', machineId: 'ST1' });

// new capabilities are opt-in, added alongside:
SOS.init({ button: '#btnSOS', machineId: 'ST1', cacheDuration: 60000, debug: true });
Why: this is called being backward compatible. Every page that integrated the widget on day one keeps working forever, even as the library grows new features nobody asked for yet.
15

The backend: Node-RED + MySQL

Backend

Node-RED lets you build an HTTP API by wiring together small visual blocks instead of writing a server from scratch. Our two endpoints are each a short chain of blocks:

GET /sos-alerts→ build SQL→ run query→ format JSON→ respond
POST /sos-alert→ parse body→ validate→ insert (or 400)→ respond

The query itself uses a prepared statement — placeholders (?) instead of pasting the user's text directly into the SQL string:

msg.topic = "INSERT INTO sosalert_history (machineid, alert) VALUES (?, ?);";
msg.payload = [machineid, alert];  // values are sent separately from the query
Why placeholders matter: if we instead wrote "...VALUES ('" + alert + "')", someone could type ' ); DROP TABLE sosalert_history; -- into the "Other" box and it would run as real SQL — a classic SQL injection attack. Placeholders keep user text as pure data, never as code.

Finally, a catch node wraps the whole flow so a database hiccup returns a clean 500 JSON error instead of leaving the browser's request hanging forever with no answer at all.

That's the whole build, one concept at a time — HTML that builds itself, CSS that themes itself, JavaScript that protects its own state, and a backend that never trusts what it's handed.
